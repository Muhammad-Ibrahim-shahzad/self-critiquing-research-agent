# Self-Critiquing Research Agent

An agentic research assistant built with **LangGraph** that doesn't just retrieve and answer — it evaluates its own output for completeness and factual grounding, and automatically retries with a refined search query when its answer falls short.

Unlike a standard RAG pipeline, this agent implements a genuine **self-correction loop**: it researches a topic, drafts an answer, critiques that answer against the actual search results, and — if the critique finds unsupported claims or missing information — refines its search query and tries again, up to a configurable retry limit.

## How it works

```
User Query → Research → Draft → Critique → [Good enough?]
                ↑                                │
                └──────────── No, retry ──────────┤
                                                   │
                                          Yes → Final Answer
```

1. **Research** — Queries the [Tavily](https://tavily.com) search API for current, relevant information on the topic.
2. **Draft** — An LLM (via [Groq](https://groq.com), `llama-3.3-70b-versatile`) writes an answer grounded in the retrieved search results.
3. **Critique** — A second LLM call evaluates the draft against two criteria:
   - **Completeness** — Does the answer fully address the question?
   - **Groundedness** — Is every claim in the answer actually traceable back to the search results, or did the model hallucinate facts, numbers, or dates not present in the sources?
4. **Decide** — If the critique flags the answer as incomplete or ungrounded, the agent loops back to research using a refined query (also generated by the critique step) — up to 2 retries, to prevent infinite loops. Otherwise, it returns the final answer.

## Why this is different from basic RAG

Standard RAG pipelines retrieve once and generate once, with no mechanism to catch their own mistakes. This agent treats its own draft as something to be checked, not trusted — closing the loop between generation and verification, and using that verification to drive real, LLM-generated corrective action (a refined search query), not just a retry of the same query.

## Tech Stack

- **LangGraph** — orchestrates the agentic control flow (state graph, conditional edges, looping)
- **Groq API** (`llama-3.3-70b-versatile`) — drafting and critique LLM calls
- **Tavily API** — web search, purpose-built for LLM agents
- **Python** (`json`, `TypedDict` for structured state and LLM output)

## Setup

1. Clone the repo and create a virtual environment:
   ```bash
   git clone https://github.com/Muhammad-Ibrahim-shahzad/self-critiquing-research-agent.git
   cd self-critiquing-research-agent
   python3 -m venv venv
   source venv/bin/activate  # Windows: venv\Scripts\activate
   ```

2. Install dependencies:
   ```bash
   pip install -r requirements.txt
   ```

3. Add your API keys to a `.env` file in the project root:
   ```
   GROQ_API_KEY=your_groq_key_here
   TAVILY_API_KEY=your_tavily_key_here
   ```
   - Get a free Groq key at [console.groq.com](https://console.groq.com)
   - Get a free Tavily key at [tavily.com](https://tavily.com)

4. Run it:
   ```bash
   python agent.py
   ```
   You'll be prompted for a query, then see each step (research → draft → critique) print live, followed by the final, verified answer.

## Example

**Query:** *"Weather in Lahore"*

The agent researched current conditions, drafted an answer, then caught itself citing a 10-day forecast that wasn't actually in the search results — flagged it as `NEEDS_IMPROVEMENT`, refined the query to `"10-day weather forecast for Lahore"`, and retried with better-grounded results.

## Design notes

- **Structured LLM output**: The critique step forces the LLM to respond in strict JSON (`verdict`, `reason`, `ref_query`) rather than free text — free-text critiques are unreliable to parse programmatically (e.g., "doesn't need improvement" still contains the substring "needs improvement"). A closed set of verdict values makes the retry decision deterministic.
- **Retry cap**: Capped at 2 retries to guarantee termination. This is a known trade-off — if the critique still finds issues after the cap, the agent returns its best-effort answer rather than looping forever. In a production setting this would be logged/flagged rather than silently returned.
- **Graceful JSON failure handling**: If the critique LLM ever returns malformed JSON, the agent defaults to `NEEDS_IMPROVEMENT` rather than crashing or silently assuming the answer is fine.
